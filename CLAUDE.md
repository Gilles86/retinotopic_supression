# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

`retsupp` ("retinotopy + suppression") is an active 7T fMRI project. Subjects perform a visual search task with a high-probability (HP) distractor location while concurrently being shown a sweeping bar PRF mapping stimulus. The science question is whether population receptive fields (PRFs) shift toward or away from the HP distractor location ("history-driven suppression"), and how this varies across the visual hierarchy. The repo is the working location for the VSS2026 talk by Dock H. Duncan and ongoing analysis with collaborators at VU Amsterdam, UZH, Granada, and Zhejiang.

Data: BIDS dataset `ds-retsupp`. 30 subjects in `retsupp/data/subjects.yml`, 2 sessions × 6 runs (a few subjects have 5 runs — see `Subject.get_runs`), TR = 1.6 s, 258 volumes per run, NORDIC denoised, fMRIprep'd.

- Local BIDS root: `/data/ds-retsupp`
- Cluster BIDS root: `/shares/zne.uzh/gdehol/ds-retsupp`

## Architecture

### Data access: `retsupp/utils/data.py`

Everything goes through `Subject(subject_id, bids_folder=...)`. Treat this class as the single source of truth for paths, file conventions, and session/run quirks. Key methods:

- `get_bold(session, run, type=...)` — `type` ∈ `{'fmriprep', 'cleaned', 'prf_regressed_out', 'raw'}`; each maps to a different derivatives folder.
- `get_stimulus(session, run, resolution)` — reconstructs the bar paradigm as a `(T, R, R)` array from the events.tsv. Per-subject FOV/bar parameters come from each run's `expsettings.yml` (the YAML path differs for `subject_id < 3` vs ≥ 3 — see `get_experimental_settings`).
- `get_onsets(session, run)` — events.tsv with `onset` re-referenced to the first scanner pulse.
- `get_hpd_locations()` — returns the HP-distractor location per (session, run), inferred from the modal `distractor_location` of feedback events. **Use this**, not the older `get_distractor_mapping_old`, which only works pre-counterbalancing fix and has a known bug for subjects 1–2 (block order `AA BBB CCC DDD A` vs intended `AAA BBB CCC DDD`).
- `get_prf_parameters_volume(model=N, type=...)` — `type` ∈ `{'mean', 'runwise', 'conditionwise'}`. Returns NIfTI images per parameter; pass `return_images=False` to get per-voxel arrays via the BOLD masker.
- `get_retinotopic_roi('V1_L', bold_space=True)` — uses neuropythy-inferred `varea` atlas under fMRIprep's freesurfer subdir. Suffix `_L`/`_R` restricts to a hemisphere.
- `get_eccentric_roi(roi='V1_L', quadrant='upper_right', ...)` — preconstructed quadrant masks under `derivatives/stimulus_rois`, generated by `eccentric_glm/get_rois.py`.
- `get_conditionwise_summary_prf_pars(model=8)` — returns the master long-format DataFrame used by attention modeling: per-voxel × per-condition PRF parameters, joined to a mean-model baseline (model 4), with rotated `(x_rotated, y_rotated)` so the distractor is always at the top, plus distance-from-distractor columns. This is the typical entry point for downstream modeling.

The module-level constants `distractor_locations` and `location_angles` at the bottom of `data.py` define the four HP locations on a 4°-eccentricity ring. Reuse them; don't redefine.

### PRF model labels

Several scripts dispatch on an integer `model_label`:

| Label | Model | Flexible HRF? | Notes |
|-------|-------|---------------|-------|
| 1 | Gaussian PRF | no | baseline |
| 2 | Difference-of-Gaussians | no | |
| 3 | Gaussian | yes | |
| 4 | DoG | yes | the **canonical "mean" model** used as baseline in conditionwise/runwise fits |
| 6 | Divisive Normalization Gaussian | yes | from braincoder `DivisiveNormalizationGaussianPRF2DWithHRF` |
| 7–11 | DoG variants | yes | parameter labels match model 4 (see `get_prf_parameter_labels`) |
| 8 | DoG variant | yes | the model used by `fit_attention_model.py` |

When adding a new model, update `get_prf_parameter_labels` *and* the dispatch in `fit_prf.py` / `fit_run.py` / `fit_condition.py`. They are not unified.

### Pipeline, in order

1. **MRI conversion** — `prepare/convert_raw_mri_data.py` (DICOM → BIDS).
2. **NORDIC denoising** — `nordic/run_nordic.m` (MATLAB), submitted via `nordic/submit_nordic_job.sh`.
3. **fMRIprep** — `cluster_preproc/fmriprep.sh` (Apptainer, runs on cluster only). Output spaces: `T1w fsnative`. `--dummy-scans 4`. Uses `cluster_preproc/bids_filter.json`.
4. **Cleaning + mean ts** — `preprocess/clean.py` (interpolates motion-outlier frames, regresses confounds → `derivatives/cleaned`), then `preprocess/mean_ts.py` builds the across-runs/sessions mean BOLD into `derivatives/mean_signal`. Both submitted by `preprocess/preprocess_subject.slurm` (SLURM array, one job per subject).
5. **PRF fits** — `modeling/fit_prf.py` (mean across runs, three-stage: grid → refine baseline/amplitude → GD), `fit_run.py` (per run), `fit_condition.py` (groups runs by HP-distractor condition, fits 4 PRF parameter sets per voxel). Each writes to `derivatives/prf{,_runfit,_conditionfit}/model{N}/sub-XX/`.
6. **Neuropythy retinotopy atlas** — `neuropythy/register_retinotopy.py` runs neuropythy to produce `inferred_varea/angle/eccen/sigma.mgz` in fMRIprep's freesurfer subdir. ROIs come from this.
7. **Eccentric / quadrant ROIs** — `eccentric_glm/get_rois.py` builds per-ROI × hemisphere × quadrant masks under `derivatives/stimulus_rois`.
8. **PRF summary tables** — `modeling/summarize_prf.py` produces `derivatives/prf_summaries[.conditionwise]/modelN/sub-XX/...prf_voxels.tsv` consumed by `Subject.get_conditionwise_summary_prf_pars` and the modeling notebooks.
9. **Eccentric GLM** — `eccentric_glm/fit_glm.py` fits a per-distractor-location GLM on PRF-regressed-out residuals (created by `eccentric_glm/clean_data.py`), then `summarize_glms.py` aggregates betas/zmaps by ROI×quadrant.
10. **Attention model** — `modeling/fit_attention_model.py` fits a 2-parameter model (`attention_sd`, log-`ratio`) per ROI per subject from the conditionwise PRF parameters using a precision-weighted product of Gaussians (4 attention Gaussians × PRF Gaussian).

The `Subject.get_bold(type='prf_regressed_out')` files are produced by `eccentric_glm/clean_data.py` — that is, the eccentric GLM operates on data with the mean PRF prediction regressed out, which isolates the residual stimulus-locked signal at HP-distractor locations.

### Data flow conventions

- Per-subject derivatives layout: `<bids>/derivatives/<analysis>/[modelN/]sub-XX/[ses-Y/[func/]] sub-XX_<...>_desc-<param>.nii.gz`. Use `Subject` accessors to assemble these — paths sometimes branch on `subject_id < 3` (early subjects had a different sourcedata layout).
- `_runfit` and `_conditionfit` derivatives reuse the **mean-fit** PRF parameters as initialization (see e.g. `fit_run.py:38-52`). Don't refit from a grid for run/condition fits.
- For all volume-space work, use the BOLD-mask masker via `Subject.get_bold_mask(return_masker=True)`. Atlases (varea, hemisphere) are resampled to BOLD space with `bold_space=True` on the relevant accessor.

### Avoid the "user env retrieval failed" dogpile

When you submit a large array (~500+ tasks) on this cluster, hundreds
of jobs will start within seconds of each other. They all then attempt
to read `$HOME/.bashrc` and the rest of the user-profile chain at the
same instant. The NFS server hosting `$HOME` can't keep up, so a
fraction of tasks fail with `user env retrieval failed requeued held`
and end up stuck pending. They don't auto-retry; you have to
`scontrol release <jobid>` them by hand.

**Primary mitigation: SLURM array throttle.** Cap how many array
tasks run simultaneously, e.g. `--array=1-1620%50` runs at most 50
concurrently. NFS profile reads are then bounded regardless of array
size. Default to `%50` for arrays >100 tasks. The cost is wall time
(throughput halves vs unthrottled), but it's the only mitigation that
actually works for arrays of 500+ tasks.

**Secondary: random startup jitter** in the SLURM script. Pre-existing
mitigation, kept as defense-in-depth even when throttled:

```bash
# Spread out NFS profile reads across a 60-second window.
sleep $(( (RANDOM % 60) + 1 ))
```

If you still see held tasks despite both mitigations, release them:

```bash
held=$(squeue --me -t PD -r --format="%i %r" | grep -i "user env" | awk '{print $1}')
echo "$held" | xargs -I {} scontrol release {}
```

The provided `submit_prf_chunked.sh` already applies `%50` throttle
by default; override via `THROTTLE=N bash submit_prf_chunked.sh ...`.

## Project-specific gotchas

- **Subjects 1 and 2** use a different sourcedata path layout (`sub-{N}/ses-{ses+1}/...` with no run subdir) and have a buggy 12-run block order (`AA BBB CCC DDD A`). Several `Subject` accessors branch on `self.subject_id < 3` — preserve this when adding new accessors.
- **Sessions are 1-indexed in derivatives but the YAML logs use `ses-{session+1}`** (because session 1 in BIDS = lab visit 2; visit 1 was a behavioral pretest). `get_experimental_settings` and `get_onsets` already handle this; replicate the same logic if you read sourcedata directly.
- `Subject.get_runs(session)` returns `[1..6]` except sub-20 ses-1 and sub-24 ses-2 (5 runs each). Always call this rather than hardcoding `range(1, 7)`.
- `cleaned` BOLD is sometimes 259 volumes; `fit_run.py` and `fit_condition.py` crop to 258. New scripts that load cleaned BOLD should do the same.
- Stimulus FOV/bar geometry varies subtly per run because `eccentricity_stimulus` and `size_stimuli` are read from per-run `expsettings.yml`. Do not assume a fixed 4° eccentricity — call `Subject.get_experimental_settings`.

## Experimental constants (typical values)

Values from `expsettings.yml` — small per-run variation; check via `Subject.get_experimental_settings` when accuracy matters.

| Quantity | Value | Notes |
|---|---|---|
| `eccentricity_stimulus` | **4.0°** | distance fovea → search-array item; the AF ring eccentricity (`ecc_distractor` in `data.py`) |
| `size_stimuli` | **1.5°** | search-item "footprint" diameter; used in aperture formula `r = ecc − size/1.8` |
| `radius_bar_aperture` | **3.17°** | bar PRF mapping aperture (= 4 − 1.5/1.8) |
| `distractor_radius` (in `Subject.get_stimulus_with_distractors`) | **0.4°** | radius of disk in the FIT paradigm. Function default; ⚠️ verify it matches the actual rendered disk size — may need to be 0.75° (= size_stimuli/2) |
| Search-array layout | 4 corner positions on a circle at 4° eccentricity, on the diagonals | Targets and distractors only ever at these 4 positions |
| Distractor location codes | 1=upper_right, 3=upper_left, 5=lower_left, 7=lower_right, 10=no distractor | `Subject.location_mapping` translates to strings |
| Target location codes | same 4 codes 1/3/5/7 | empirically target NEVER co-occurs with the trial's distractor at the same location |
| Target/distractor on-window | ~1.5s (`max_distractor_duration`) | from target onset to feedback (capped) |
| TR | 1.6 s | |
| Volumes per run | 258 | (cleaned BOLD is sometimes 259; crop to 258) |

The 4 ring positions in degrees (use these — don't redefine):

```python
from retsupp.utils.data import distractor_locations  # noqa
# distractor_locations['upper_right'] == ( ecc/√2, +ecc/√2)  with ecc=4
# Always use the underscore form 'upper_right' — the legacy space form
# 'upper right' exists in some old code; underscore is canonical for
# condition_pars and the AF model channel order.
```

## Conventions

- All fitting scripts take a positional `subject` (int) and `--bids_folder`; SLURM array index `$SLURM_ARRAY_TASK_ID` provides the subject ID, zero-padded for the directory pattern `sub-{N:02d}`.
- Plotting prefers seaborn (per global instructions). For per-condition panels, use `FacetGrid`/`relplot`/`catplot`. For polar/rotated visualizations of the conditionwise PRFs, use the rotated `(x_rotated, y_rotated)` columns from `get_conditionwise_summary_prf_pars` — this puts every condition's distractor at the top, so attention effects average meaningfully across conditions.
- Distractor locations are encoded with two parallel conventions: the integer codes `1.0/3.0/5.0/7.0/10.0` from the events.tsv (`Subject.location_mapping` translates) and the string keys `'upper_right'/'upper_left'/'lower_left'/'lower_right'` used by the conditionwise PRFs and the attention model. The string form (with underscore) is the canonical form for `condition_pars`; the space-separated form (`'upper right'`) appears in some legacy code (`distractor_locations` dict in `data.py`). Watch for this when joining data.
